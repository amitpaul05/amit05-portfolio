We are currently in the middle of refactoring a legacy system, and today we finally got access to the existing production database dump. I started with the restore process expecting it to work out of the box, but while testing the user creation flow I hit a django.db.utils.IntegrityError, which indicated something was wrong at the database layer rather than the application logic.

After tracing the schema state, I found that the exported SQL (from TablePlus) had all primary key columns defined as bigint NOT NULL without any attached sequences or DEFAULT nextval(...). This effectively broke auto-increment behavior across the tables. The initial assumption was a partial corruption during export, so I attempted to patch the schema using sed, converting affected fields into bigserial/serial.

Later during deeper inspection, I discovered a proper PostgreSQL custom-format .dump was also available, which preserved sequence definitions and setval statements correctly. At that point, I switched the restore strategy to pg_restore instead of manual SQL fixes.

Finally, I structured the production setup by moving DB initialization scripts into Docker (/docker-entrypoint-initdb.d) and aligned environment separation for local vs production (DB host, Redis/Celery brokers, DEBUG flags, and CORS/CSRF configs).
